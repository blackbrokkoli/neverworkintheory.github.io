--- layout: post author: Geoff Cramer title: "A case history analysis of software error cause-effect relationships" date: 2020-02-21 categories: ["Development Process", "Program Analysis"] ---

[Nakajo1991] analyzes ~700 software errors in 4 commercial measuring-control products and utilizes a restricted version of fault tree analysis (FTA) to capture meaningful cause-effect failure patterns. It also discusses the effectiveness of Structured Analysis and Structured Design (SA/SD) methods on repressing the identified error mechanisms.

The study observes failures at chosen points along its “cause-effect process of software errors” (chosen points in bold):

  1. inappropriate management of work system
  2. work system flaw (process, individual, environmental flaws)
  3. human error
  4. program fault
  5. system failure

The methodology provides a systematic mapping of software failures to their ultimate cause and leverages statistical analysis to draw conclusions about rates of particular errors across projects. However, it is limited in the types of software it analyzes (2 system products and 2 firmware products) and was not able to study certain paths along its own defined error cause-effect process (namely individual and environmental flaws).

[Nakajo1991] concluded that:

  • Two major error patterns resulted from hardware or software interface misunderstandings.
  • Two more common error sources emerged from system and module function misconceptions.
  • SA/SD efforts to mitigate these failures were effective on the first two but not the last two error-occurrence sources.

    • [Nakajo1991] Takeshi Nakajo and Hitoshi Kume: "A case history analysis of software error cause-effect relationships" IEEE Transactions on Software Engineering ( Volume: 17, Issue: 8, Aug 1991) 10.1109/32.83917

    Approximately 700 errors in four commercial measuring-control software products were analyzed, and the cause-effect relationships of errors occurring during software development were identified. The analysis method used defined appropriate observation points along the path leading from cause to effect of a software error and gathered the corresponding data by analyzing each error using fault tree analysis. Each observation point's data were categorized, and the relationships between two adjoining points were summarized using a cross-indexing table. Four major error-occurrence mechanisms were identified; two are related to hardware and software interface specification misunderstandings, while the other two are related to system and module function misunderstandings. The effects of structured analysis and structured design methods on software errors were evaluated.< >
